1) Project Summary
------------------
Project name: NovaShield AI-Powered Next-Generation Firewall

What it is:
- A security gateway in front of a backend banking API.
- It inspects incoming traffic using layered defenses:
  1. IP blacklist
  2. WAF regex attack detection
  3. per-route rate limiting
  4. JWT authentication and admin RBAC
  5. AI threat classification service
- It forwards safe traffic to backend and blocks malicious traffic.
- A React dashboard streams live logs, counters, events, and blacklisted IPs.

Main value proposition:
- Combines rule-based security + ML-based threat detection + observability in one full-stack system.


2) High-Level Architecture
--------------------------
Client/User

      -> Backend (Rust, Axum) [port 8081]

Runtime services in docker-compose:
- redis
- ai_engine
- backend

Gateway is the security brain:
- receives all API traffic
- validates/blocks/forwards
- records telemetry
- pushes live events over WebSocket


3) Request/Data Flow (Exact)
----------------------------
3.1 Login flow (/api/login)
- Client sends username/password to gateway.
- Gateway extracts client IP and creates request context with request_id.
- Gateway runs security::inspect_common:
  - blacklist check
  - WAF pattern check on path/body
  - login route rate limit check
- If rejected -> 403/429 with structured error.
- If passed -> gateway calls AI engine /predict.
- AI decision:
  - block => gateway returns 403 AI_BLOCKED
  - allow/fallback => forwards request to backend /login
- Backend issues JWT token (HS256) and returns token + account id.
- Gateway logs decision and latency.

3.2 Protected routes (/api/balance, /api/transfer)
- Same checks as above + JWT verification at gateway before forwarding.
- For /transfer, gateway inspects request body for WAF payloads.
- If JWT invalid => 401
- If user is valid => forwarded to backend.

3.3 Admin routes
- /api/admin/snapshot
- /api/admin/logs
- /api/admin/metrics
- /api/admin/blacklist/add
- /api/admin/blacklist/remove

All admin routes require valid JWT + role=admin (enforced in gateway verify_admin_jwt).

3.4 Dashboard live flow
- Dashboard loads snapshot from /api/admin/snapshot.
- Opens WebSocket /ws/live.
- Receives event stream:
  - snapshot
  - request_log
  - security_event
  - system_notice
- UI charts and counters update in real-time.


4) Security Layers and How Attacks Are Prevented
-------------------------------------------------
4.1 IP Blacklist
- Stored in gateway in DashMap.
- Seeded from config on startup.
- Can be changed at runtime through admin endpoints.
- If IP is blacklisted -> request blocked (403 BLOCKED_IP).

4.2 WAF Rules (Regex)
Implemented in gateway/src/security.rs via WafEngine.

Rules:
- SQLi:
  - (or/and ... = ...)
  - union select, drop table, sleep(), --
- XSS:
  - <script, onerror=, onload=, javascript:
- Path traversal:
  - ../, /etc/passwd, boot.ini
- Command injection:
  - | cat/sh/bash, ; curl/wget, && cat/curl/wget

When matched:
- Gateway returns 403 with rule-specific code (WAF_SQLI / WAF_XSS / WAF_TRAVERSAL / WAF_COMMAND).
- Security event recorded.
- For critical WAF rule types, IP may be promoted to blacklist automatically.

4.3 Rate Limiting
- Sliding fixed window per route key for 60 seconds.
- Route-specific limits from config:
  - login_per_minute = 5
  - transfer_per_minute = 20
  - balance_per_minute = 120
  - default_per_minute = 600
- If exceeded -> 429 RATE_LIMITED with headers:
  - Retry-After
  - X-RateLimit-Limit
  - X-RateLimit-Remaining

4.4 JWT Auth + RBAC
- JWT algorithm: HS256
- Gateway validates token for protected routes.
- Admin routes require role="admin".
- Backend assigns admin role to usernames: admin, root, superadmin.

4.5 AI Threat Enforcement
- Gateway sends metadata (ip, path, method, user_agent) to AI engine /predict.
- AI engine predicts label from model.
- Decision rule:
  - label == BENIGN => allow
  - else => block
- Gateway behavior:
  - block => 403 AI_BLOCKED
  - allow => continue
  - AI unavailable/timeout/invalid response => fallback allow (fail-open, availability-first)


5) AI/ML Details (Very Important for Viva)
-------------------------------------------
5.1 Model type
- RandomForestClassifier (scikit-learn)

5.2 Training dataset
- CICIDS2017 derived data under ai_engine/model/dataset
- cleaned dataset file: ai_engine/model/dataset/cleaned.csv
- raw day-wise CICIDS files are also present.

5.3 Feature handling
- Training uses all columns except Label.
- Inference feature order saved in features.pkl.
- Runtime aligns input vectors exactly to saved feature list.

5.4 Preprocessing
- Replace inf/-inf with 0
- Drop NaN rows
- Trim column names
- Label normalization for inconsistent web attack strings
- LabelEncoder encodes text labels
- StandardScaler scales feature matrix

5.5 Train/test split
- train_test_split with:
  - test_size=0.2
  - random_state=42
  - stratify=y

5.6 Random Forest hyperparameters
From ai_engine/model/retrain_model.py:
- n_estimators = 120
- max_depth = 20
- random_state = 42
- n_jobs = 1
- class_weight = "balanced_subsample"

5.7 Saved artifacts
- model.pkl
- scaler.pkl
- features.pkl
- label_encoder.pkl

5.8 Inference output
- label (class name)
- confidence (if predict_proba available)
- predicted_class (optional)
- gateway uses decision = allow only when label BENIGN else block.

5.9 AI API endpoints
- GET /health
- POST /predict
- POST /predict/batch
- docs: /docs

5.10 Additional feature extraction module
- ai_engine/feature_extractor.py can sniff packets via Scapy and build flow features.
- Uses timeout/packet trigger and optional rate guard.
- This module is useful for packet-flow simulation and offline experiments.
- Current gateway->AI path uses request metadata, not live packet sniffing directly.

12) Important Design Choices (Interview Talking Points)
--------------------------------------------------------
- Rust chosen for gateway/backend for performance and memory safety.
- Rule-based + AI hybrid gives both explainability (WAF rules) and adaptive detection (ML).
- Shared Rust crate prevents payload schema drift between services.
- Fail-open on AI timeout preserves service availability.
- WebSocket + Prometheus provide real-time and historical observability.
- Config-driven controls (rate limits, blacklist, AI toggle, timeouts).


13) Known Limitations / Honest Answers for Jury
------------------------------------------------
- AI service currently uses pre-trained classical ML, not online continuous learning.
- AI input from gateway is metadata-only; full packet feature extraction path is separate module.
- Redis rate-limit helper exists but in-memory limiter is primary path in current gateway flow.
- Backend is a mock banking API with in-memory balances (non-persistent).
- JWT uses shared secret (HS256); production may prefer key rotation/strong secret management.
- WAF regex signatures can have false positives/false negatives and need tuning.


14) Security Hardening Suggestions (If Asked)
---------------------------------------------
- Enforce HTTPS only and HSTS.
- Rotate secrets via vault/KMS and environment injection.
- Persist balances in DB with transaction integrity.
- Add mTLS/service auth between gateway and AI engine/backend.
- Add request signing and stricter admin endpoint audit trails.
- Add model drift monitoring and periodic retraining pipeline.
- Add canary mode for new WAF/AI rules before hard block.


15) Quick Viva Q&A Cheat Sheet
------------------------------
Q: Which ML model is used?
A: RandomForestClassifier with n_estimators=120, max_depth=20, class_weight=balanced_subsample, trained on CICIDS2017 cleaned dataset.

Q: How does AI block work in gateway?
A: Gateway calls AI /predict; if returned label is not BENIGN, gateway responds 403 AI_BLOCKED and logs a critical security event.

Q: What if AI engine is down?
A: Gateway uses fail-open fallback allow (AiDecision::Fallback) to prevent single-point-of-failure outages.

Q: How do you stop SQL injection/XSS?
A: WAF regex inspection on path and body before proxying. Matching payload is rejected with 403 and event logged.

Q: How is rate limit implemented?
A: Per-IP, per-route 60-second window in gateway DashMap; route limits from config. 429 includes retry-after and limit headers.

Q: How do you monitor system health?
A: Admin snapshot/log endpoints, Prometheus metrics endpoint, and live WebSocket event stream consumed by React dashboard.

Q: What are main technologies used?
A: Rust (Axum/Tokio), Python (FastAPI/scikit-learn), React+Vite+Recharts, Redis, Docker Compose.
